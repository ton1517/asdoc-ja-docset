クラスpublic class DataStore
継承DataStore EventDispatcher Object

言語バージョン: ActionScript 3.0
製品バージョン: Adobe Digital Enterprise Platform Data Services for Java EE 3
ランタイムバージョン: Flash Player 9, AIR 1.1

DataStore は、1 つ以上の DataService インスタンスのトランザクションをまとめたものです。サーバーに対する保留状態の変更を管理し、サーバーから受信した競合も格納します。デフォルトでは、関連付けを管理していない DataService はそれぞれ、独立した DataStore を自動的に取得します。commit() メソッドを呼び出すたびに、その DataService に対して加えられた変更のみが送信されます。

管理対象の関連付けを使用すると、同じチャネルセットを共有し、同じ値の use-transactions フラグを持つすべての DataService インスタンスの間で同じ DataStore が共有されるように、デフォルト値が変更されます。DataStore を共有している場合、commit() メソッドを 1 回呼び出すだけで、その DataStore を共有する全 DataService インスタンスによって加えられたすべての変更がコミットされます。管理対象の関連付けを使用している場合、複数の宛先で行われた変更の間に依存関係が存在する可能性があり、サーバーにメッセージを送信するときにクライアントで操作の順序を維持することが重要になるので、デフォルト値が変更されます。

デフォルト値を変更する必要がある場合は、独自の DataStore を作成し、それを 1 つ以上の DataService インスタンスの dataStore プロパティに手動で割り当てます。この作業は、fill や getItem などの他の DataService API 呼び出し発行する前に、DataService を最初に初期化する際に行います。メモ：同じ DataStore を共有するすべての DataService で、同じチャネルセットおよび同じ設定の useTransactions フラグが使用されている必要があります。設定が一致していない場合はエラーがスローされます。

commitQueueMode

プロパティ 
commitQueueMode:int

言語バージョン: ActionScript 3.0
製品バージョン: Adobe Digital Enterprise Platform Data Services for Java EE 3
ランタイムバージョン: Flash Player 9, AIR 1.1

最初にコミットしたバッチに対する応答をクライアントが待っている間に 2 番目のバッチに対する commit を呼び出した場合に、2 番目にコミットされた変更バッチをいつサーバーに送信するかを制御します。最初のバッチをコミットしたときにサーバーによって加えられる変更に依存する変更が 2 番目のバッチに含まれている場合、そのバッチを直ちにサーバーに送信すると問題が発生する発生する可能性があります。例えば、最初のバッチでサーバーが作成するアイテムを更新するような場合です。サーバーが割り当てる ID をクライアントがまだ受け取っていない場合、更新にはそのアイテムの適切な ID が含まれません。最初のバッチが完了するまで 2 番目のバッチを送信しないように設定すると、時間に依存する一連の変更（カーソル位置を繰り返し更新する場合など）を実行するために必要な時間が長くなる場合があります。

デフォルトのモードは CQ_AUTO です。このモードでは、現在処理中のバッチと次にコミットするバッチの間に単純な依存関係があるかどうかが確認されます。依存関係がないと見なされた場合は、新しいバッチが直ちに送信されます。現在処理中の作成操作に依存する更新操作や削除操作がある場合は、作成が完了するまで更新や削除が保留されます。この設定にすると、依存関係の確認が行われるため、コミットのたびに若干のオーバーヘッドが発生しますが、通常は、他の 2 つのモード（CQ_NOWAIT と CQ_ONE_AT_A_TIME）の利点をうまく折衷できます。

CQ_NOWAIT の場合は、依存関係を確認せずに 2 番目のバッチが直ちに送信されます。クライアント上で新しいオブジェクトに ID を割り当て、パフォーマンスを上げるためにバッチをオーバーラップさせる必要がある場合は、このモードを使用することを推奨します。

CQ_ONE_AT_A_TIME の場合は、前のバッチが完了してから新しいバッチがコミットされます。このモードの場合、前のバッチの処理によるサーバー側での変更がまだ完了していない状態で次のバッチがサーバーに送信されることがないので、エラー処理のロジックは簡潔になります。また、いずれかのバッチの処理でエラーが発生した場合、そのエラーがクライアント側で解決されるまで、次のバッチはサーバーに送信されません。

デフォルト値： CQ_AUTO。

commit

()メソッド
public function commit(itemsOrCollections:Array = null, cascadeCommit:Boolean = false):mx.rpc:AsyncToken

言語バージョン: ActionScript 3.0
製品バージョン: Adobe Digital Enterprise Platform Data Services for Java EE 3
ランタイムバージョン: Flash Player 9, AIR 1.1

この DataStore によって管理されているコレクションおよびアイテムに対する保留状態の変更をコミットします。autoCommit プロパティが true であるときはトランザクションが使用されないので、このメソッドを呼び出す必要はありません。トランザクションを使用するには、autoCommit プロパティを false に設定し、このメソッドを直接呼び出して変更のバッチをコミットします。通常は、引数を指定せずに「commit()」を呼び出し、保留状態のすべての変更をコミットします。オプションの引数を指定すると、保留状態の変更の一部のみをコミットできます。

DataStore 内のアイテムに対する変更をコミットするには、先に未解決の競合をすべて解決する必要があります。

アセンブラーで use-transactions が true に設定されている場合は、すべての変更がアトミックにコミットされます。バッチで指定された操作が 1 つでも失敗した場合は、すべての操作が適用されません。use-transactions が false に設定されている場合は、障害が発生するまで各操作が個別にコミットされ、障害が発生した時点で処理が停止します。エラーの前に発生した変更はすべて適用され、エラーの後に発生した変更は適用されません。バッチ処理でエラーが発生した場合は、すべての変更が未コミットのキューに戻されます。エラーの原因になった変更を削除する場合は、revertChanges を呼び出す必要があります。

デフォルトでは、commit メソッドを呼び出すと、この DataStore に対して作成、変更、および削除されたすべてのアイテムに対する保留状態の変更がすべてコミットされます。これらの変更の一部のみをコミットする場合は、バッチに含める変更を定義する管理対象 ArrayCollection インスタンスと管理対象アイテムの組み合わせを指定できます。指定した配列コレクションごとに、その配列コレクション内のアイテムのメンバーシップまたは順序に対する変更、またはその配列コレクション内のアイテムに対する変更がすべてコミットされます。また、個々の管理対象アイテムのリストを指定して、それらのアイテムに対する変更のみをコミットすることもできます。管理対象のコレクションまたはアイテムではないオブジェクトを itemsOrCollections パラメーターで指定すると、DataServiceError がスローされます。

itemsOrCollections パラメーターを使用していて、管理対象アイテムが他のアイテムを参照する関連付けパラメーターがある場合は、cascadeCommit パラメーターを使用して、関連付けられているアイテムに対する変更もバッチに含めるかどうかを制御できます。例えば、items パラメーターで Group アイテムを指定し、Group インスタンスに Person インスタンスのリストを参照する members プロパティがある場合、cascadeCommit を true にすると、Person インスタンスに対する変更もコミットされる変更のセットに含まれます。cascadeCommit が false の場合は、Group に対する変更のみが含まれます。従属するすべての変更をバッチに含めるには、cascadeCommit=true を使用します。

パラメーター

itemsOrCollections:Array (default = null) — これはオプションのパラメーターであり、保留状態のすべての変更をコミットするときはデフォルトで null になります。保留状態の変更の一部のみをコミットする場合は、この引数を使用して、管理する ListCollectionView インスタンスと管理するアイテムのリストを指定します。ほとんど場合、ListCollectionView オブジェクトは、fill メソッドに渡した ArrayCollection です。このメソッドに適したアイテムは、各アイテムの管理対象バージョンです。これらは、getItem または createItem から取得されるアイテム、または getItemAt メソッドを使用して管理対象コレクションから取得されるアイテムです。この配列内のいずれかの値で定義されているアイテムの変更のみがコミットされます。
 
cascadeCommit:Boolean (default = false) — true の場合は、このリストで指定された関連アイテムに対する変更もコミットされます。

戻り値
mx.rpc:AsyncToken — ResultEvent.RESULT または FaultEvent.FAULT の call プロパティで返される AsyncToken です。このオブジェクトにはカスタムデータを添付することができ、そのデータは後のイベント処理段階で検査できます。関連するアイテムが変更されていない場合は、AsyncToken の代わりに null が返されます。

createBatch

()メソッド 
public function createBatch(itemsOrCollection:Array = null, cascadeCommit:Boolean = false, properties:Object = null):MessageBatch

言語バージョン: ActionScript 3.0
製品バージョン: Adobe Digital Enterprise Platform Data Services for Java EE 3
ランタイムバージョン: Flash Player 9, AIR 1.1

この操作で指定されている変更を取り、新しい MessageBatch を作成して、そのバッチを uncommittedBatches リストに格納します。itemsOrCollection=null の場合は、currentBatch 内の全メッセージが新しいバッチに追加され、currentBatch として新しいバッチが作成されます。一連の変更を分割し、1 つの単位としてコミットしたり元に戻したりできる作業のブロックを指定する必要がある場合は、このメソッドを呼び出します。例えば、親オブジェクトを作成し、その親に関連付けられた子を作成する場合は、createBatch() を呼び出してこれら 2 つの変更を 1 つのグループにまとめます。これを 2 回行う場合は、それぞれ 1 つの親と 1 つの子を作成する 2 つのバッチを作成します。その後、バッチを作成した順序どおりコミットする限りは、各グループの変更を個別にコミットできます。また、最後のバッチは元に戻せますが、最初のバッチは変更されないままになります。各操作の後に createBatch(null) を呼び出すだけで、簡単なマルチレベルの「取り消し」システムを実装できます。

MessageBatch.properties プロパティを含む一連のユーザー定義プロパティで（またはこのメソッドの properties 引数で）バッチを指定できます。この情報はオフライン機能を使用するときにバッチと共に保存および復元され、クライアント上のバッチの識別に使用できます。ここで設定する値は、直列化可能である必要があります（つまり、[RemoteClass] を持つオブジェクトまたはクラス）。

itemsOrCollection パラメーターの値を指定すると、それらのアイテムまたはアイテムのコレクションに対する変更をのみを含む新しいバッチが作成されます。cascadeCommit を true に設定すると、関連付けられているアイテムに対する変更も、新しいバッチに分割されたメッセージのセットに追加されます。

パラメーター

itemsOrCollection:Array (default = null) — アイテムの配列または管理対象コレクションの配列です。アイテムの配列の場合は、各アイテムに対する変更メッセージが新しいバッチに分割されます。管理対象コレクションの配列の場合は、各コレクション内の各アイテムに対する変更が新しいバッチに分割されます。null の場合は、currentBatch 内の全メッセージが新しいバッチに追加され、currentBatch として新しいバッチが作成されます。
 
cascadeCommit:Boolean (default = false) — true の場合、関連付けられているアイテムに対する変更も、新しいバッチに分割されたメッセージのセットに追加されます。
 
properties:Object (default = null) — バッチをマークするためのユーザー定義のプロパティです。

戻り値
MessageBatch — 新規作成された MessageBatch です。

パラメーター

copyStillManagedItems:Boolean (default = true) — このデータサービスを解放した後に、別のデータサービスからの管理対象関連付けの参照が、これらの管理対象オブジェクトの一部への参照を保持している可能性があるかどうかを示します。その場合は、このデータサービスの管理対象コレクションの一部が、現在もアクティブに管理されている可能性があります。このフラグに true を渡した場合、これらの参照は削除（clear=true）または複製（clear=flase）されます。このフラグに false を渡した場合、現在アクティブな参照はそのまま保持されます。false を使用すると、システムの状態を最もすばやくクリアできます。true を使用すると、このデータサービスから取得したコレクションまたはアイテムへの参照に、管理対象になる状態が存在しなくなります。
